Changing Course

In some form, I feel like using textbooks instead of a more "friendly" resource may be to my detriment; they are, after all, intended for undergrad students. With that in mind, my frustration with them so far is understandable, because although I'm certain I could take the time to understand every single line of it, I tend to be impatient. These would take entire semesters to go through, and the C textbook in particular assumes little to no C knowledge while giving a lot of information sprinkled in-between stuff I already know. And the Signals and Systems textbook, while I can understand the math portions, it feels a bit overwhelming for certain. Basing writing my blog off these textbooks and my progress into them has been kind of successful but also quite frustrating and is making me prone to putting them off.

Hm. After a little thinking, all this jumping around I do, I'm not all that happy with it. I'm certain we've all been there before: we try to learn something by picking it up, but when we get frustrated, it's easy to try and change course. I know I'm passionate about this and all, but I'm a high school student trying to study this in my free time, not pursuing a degree in it.

I know that it's not wise to expect immediate results, and that I should be enjoying the process, but structure has never been one of my strong suits. I have ADHD to thank for that partially, but I'd rather not use it as an excuse when I know that I can do better than this. So let's see: where did I go wrong?

For one, I used a general purpose textbook in signal processing with a lot of math, intended for undergraduates. The other textbook I used is teaching a programming language that I haven't written much. I'm trying to incorporate it into a framework that doesn't even use that language. There's no use of Python or Java in these textbooks or in any of these frameworks, which are languages I'm most proficient in. I think the issue is clear: I'm overextending way too early.

Does that make the previous work I've done on these blogs useless? I don't think so. Although I'm always embarrassed to backtrack and say "I don't think I should've started this way" and try to take a different route, because I feel that reflects poorly on my ability to pursue self-motivated learning, something that assures me is that I'm still learning how to learn. Teaching myself new subjects is a skill that can be improved upon, and I've spent my life so far relying on the structure of school to keep me ahead, trusting that they knew what they were doing. At this point, I do not know what I am doing yet, and that is okay as long as I don't stay here.

So what should I change? Firstly, let's back off a bit and put off the textbooks for later. Secondly, I want to narrow down my scope. Keep in mind that my original plan was to learn general signal processing before diving into audio; with hindsight, that wasn't a good idea because when I wanted to specify in audio, it just gave me extra obligation I didn't want or need. Plus, what if I didn't want to do something signal processing related one day? Having a textbook with math I feel obligated to cover would distract and drain.

So what is my scope exactly? I shall specify in creating audio plugins, and I would like to learn the JUCE framework in order to do that. There, easy. Only took me a week.

How can I learn it? Well, there's plenty of options. There's plenty of example projects, tutorials both official and unofficial, C++ books I can read into so I can learn the language well enough, etc. But I think the key is to use each of these resources carefully and when we need them.

Setting Up JUCE

There are plenty of resources for this that I'll be studying in my free time, but for now let's get it set up. Luckily, the setup process on Linux is unusually easy. All you need to do is download the Projucer application from their website, which comes standalone alongside the JUCE assets, and boom! We're pretty much set up!

Except for one unfortunate detail: I'd like to use Visual Studio Code as my IDE, and it for some reason doesn't recognize that. It allows XCode and Visual Studio, but not VSCode. Fortunately, there is an option for a Linux makefile, which is unfortunately unfamiliar to me but I'm willing to learn, especially given that they are used fairly often.

This is trivial to get around, though; all you need to do is find the project location and open that in Code. But I'm not exactly sure how to find the example projects. JUCE has an examples folder, but it only contains the header files when the project itself shows me that I have more than that (found out later that's for the DemoRunner executable). So perhaps it's time to check the website for where to have an easy starting point.
