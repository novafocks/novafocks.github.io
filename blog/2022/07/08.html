After looking into it, I found it was too daunting to just jump into learning the full stack, so this blog will only introduce the concepts for that field of development, while actually creating an application with the UI technology.

There's a few main options for approaching the UI side: using the React library, the Angular framework, or Vue. Obviously there's many, many, more but I'd prefer to go with a popular, mainstream option because they're tested in the field extensively and have plenty of documentation.

Angular is built and maintained by Google, and I've heard that it can be opinionated and complicated, but still incredibly useful, particularly for complex applications. I don't really need to create anything complex for right now, it might be good to learn eventually, but I'll pass for right now.

Vue is newer and built by an ex-Angular employee, and I've heard that it's particularly pleasant to work with. This does sound like a good choice, but let's briefly cover the option which I'll actually be going with.

I want to learn React initially. This is because it is largely unopinionated, extremely flexible, extensible, and generally the most used choice in the industry (at least I've heard). So why not go with something so clearly reliable?

So how does full stack development work, and what tools would I use? Well, React works on the front-end of web development, which involves what the user will be seeing. There's also the back-end of web development which organizations need to worry about, managing databases and information behind the scenes. Full-stack development involves working on both of these at the same time to create a coherent and beautiful web application, often with many complex moving parts.

As for the tools, I think it would be good to start with the tried-and-true Node.js, Express.js, and MongoDB combination. MongoDB is going to be our database tool, which uses its own JSON-like formats, different from relational databases. Node.js will work on the application side and interact with our databases. Express.js will provide the web application features on top of Node.

This tech stack is called the MERN stack, which is an acronym that includes each of the individual components. There's also the popular MEAN stack, which uses Angular instead of React, but this is just my personal preference for right now.

Creating a Basic React Application

So let's create a basic application using React! Luckily, there's this awesome tool called Create React App which we can use to run a command and automatically create a template, so we'll be installing that once we have Node up and running.

I installed nvm as a version manager for Node.js rather than install the library directly, since I find that approach to be cleaner, and installed the latest LTS version: v16.16.0. Luckily, you can just run the command nvm install --lts to get the latest LTS version! That's quite convenient!

So after running the command to create a React app (npx create-react-app [name goes here]), I found that it even initializes a Git repository, complete with a .gitignore file up and ready to go, which is awesome.

I just want to take a second to appreciate the Material Icons extension for Visual Studio Code. They just add so much despite changing so little. Look at how much visual feedback those little images give!

Sorry, I've just always been a sucker for details like that. Anyways, as you can see, there's a fairly basic folder structure generated with the React app. The node_modules folder stores the... well the Node modules, the public folder seemingly holds some metadata, and the src folder holds the application and the underlying JavaScript that composes the app. The src folder is where we're going to be coding, deducing from the documentation.

Examining the App.js file, which is the one the live server tells you to edit, I notice a few interesting things. It imports a logo and some CSS, but from there it's just a function that returns some HTML code which is displayed on the webpage. I think I've heard of this before, that this is called JSX, which is just an extension to JavaScript which allows the embedding of HTML straight into the code, allowing you to input any information you want (correct me if I'm wrong, of course).

The index.js file simply calls the App function from App.js and renders it through the virtual DOM. Of course, there's a few other JavaScript files that perform other functions, like reporting the vitals and what I assume to be utilities for debugging, but that's it.

Looking in the package.json in the root folder, the start utility that npm uses calls a utility called react-scripts, and the documentation tells me that this is the main way the app reacts with the rest of the software. So if you need to update your app, it says that you mainly need to worry about that package. Neat!

So how would you implement this in a real world application? Well, the cool thing is that, as the website continually mentions, React can be used as little or as much as you need within a project; it can be used to create a couple elements of the UI, or you can use it as the basis for your entire project.

Reading the documentation, the way in which you would implement this is as simple as the explanation: you can create components or elements of your website using React, piece by piece, and implement them into your project however you like. For example, if you wanted to add a button created using React, you could create a Button.js file in a folder about UI components, and implement that into an existing website using script (couldn't put in HTML valid code because that kinda creates a mess). Of course, when using the Create React App command, React forms the basis for the app you chose and runs standalone, but either method works!
